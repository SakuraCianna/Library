# 个人知识库桌面应用设计文档

## 1. 产品定位

本应用是一个本地优先的个人知识库桌面端工具，第一版优先支持 Windows，架构上预留未来扩展到 macOS。它面向长期工作和学习场景，用户可以把 Word、PDF、XLSX、Markdown 等资料放进真实磁盘文件夹中，由应用进行解析、结构化、索引和智能问答。

核心目标不是做一个封闭的知识平台，而是做一个长期可控、可迁移、可备份的知识工作台。真实文件夹和 Markdown 笔记是用户资产主体，应用本地数据库和索引用于增强检索、问答、权限控制、回收站和结构化理解。

## 2. 已确认的核心决策

- 产品形态：Windows 桌面应用，后续可扩展到 macOS
- 数据策略：本地优先，未来可选同步
- 资产模型：真实磁盘文件夹为主，应用元数据和索引为辅
- 文件夹组织：用户可建立多个知识空间，例如 `面试`、`SpringBoot`、`工作项目A`
- Agent 范围：默认当前文件夹，可切换当前文件、当前文件夹、全库
- Agent 权限：用户可选只读、请求批准、完全访问；完全访问默认关闭，需要显式开启
- 权限隔离：文件夹默认权限和当前聊天权限分开，当前聊天可临时升到请求批准，但必须显式确认
- 结构化产物：同名 Markdown 笔记 + 文件夹 `README.md` 总览 + 本地知识块索引
- Markdown 策略：生成后就是普通文件，用户可自由修改，系统不做标记区，也不自动覆盖
- 文件变更：默认只提示，可手动开启自动解析
- 回收策略：第一版只做简单回收站机制，不做完整版本管理
- AI 模型：云端模型第一版仅接入 DeepSeek `deepseek-v4-flash`
- 本地 OCR：第一版纳入范围，按 GPU/CPU 分层执行
- 表格理解：第一版支持结构还原、语义理解和自然语言问答，不做复杂报表仪表盘
- 同步：第一版不做账号和实时同步，但支持备份、导出、导入，并预留未来同步字段
- 界面语言：应用内可见界面默认中文；文件扩展名、模型 ID、真实技术名、用户文件名保留原样
- 技术架构：Tauri v2 + React/Vite/TypeScript + Rust 核心 + Python 文档解析 Sidecar
- 本地数据库：SQLite 负责元数据、任务状态、权限、回收站和全文检索
- 本地向量数据库：LanceDB 负责向量索引和语义检索，不使用云端向量数据库

## 3. 推荐方案

采用“真实文件夹 + 元数据索引双轨型”。

真实磁盘文件夹是长期资产主体，原文件和 Markdown 笔记可直接被用户打开、备份和迁移。应用内部维护 SQLite 数据库、向量索引、表格结构、解析任务、权限、回收站和聊天引用，用于提供 AI 问答、知识块删除、OCR、表格深度理解和检索。

这个方案兼顾可控性和智能能力。即使用户未来不用这个应用，文件和 Markdown 仍然可用；应用存在时，则提供结构化知识管理和智能助手能力。

## 4. 技术架构选型

最终架构锁定为 Tauri v2 + React/Vite/TypeScript + Rust 核心 + Python 文档解析 Sidecar。

### 4.1 桌面端和前端

- 桌面壳：Tauri v2
- 前端框架：React + Vite + TypeScript
- 样式组织：组件级样式优先，避免把业务样式堆进全局 CSS
- 目标平台：第一版 Windows，架构预留 macOS

选择 Tauri v2 的原因是它更适合本地文件、权限、任务队列和长期桌面应用。Rust 层可以统一处理文件系统访问、数据库写入、权限校验和后台任务，前端只负责展示和发送明确请求。

### 4.2 Rust 核心职责

Rust 核心负责：

- 文件夹扫描和文件状态跟踪
- SQLite 元数据读写
- LanceDB 向量索引调用
- 权限拦截和高风险操作确认
- 任务队列调度
- 回收站和恢复
- 备份、导出、导入
- 与 Python Sidecar 通信
- 与 DeepSeek API 通信

所有会修改文件、索引、权限或回收站的操作都必须经过 Rust 核心。前端不能直接绕过权限写文件或写数据库。

### 4.3 本地数据库

元数据数据库选择 SQLite，本地存储在用户知识库或应用数据目录中，不使用云数据库。

SQLite 负责：

- 知识库和文件夹配置
- 文件路径、哈希、修改时间、解析状态
- Markdown 笔记状态
- 知识块元数据和来源位置
- 文件夹默认权限和聊天会话权限
- 回收站记录
- 解析任务状态
- 表格结构索引
- 聊天引用记录

全文搜索优先使用 SQLite FTS5。这样关键词搜索和元数据查询都在本地完成，不依赖网络。

### 4.4 本地向量数据库

向量数据库选择 LanceDB，本地 embedded 模式运行，不使用云端向量数据库。

LanceDB 负责：

- 知识块向量
- Markdown 笔记向量
- 表格语义摘要向量
- 当前文件、当前文件夹、全库范围内的语义检索
- Agent 回答时的候选知识召回

向量库和 SQLite 都保存稳定 ID，SQLite 保存业务状态，LanceDB 保存向量和可召回字段。删除知识块时，SQLite 先标记不可用，再同步更新 LanceDB 的可检索状态，避免已删除内容继续出现在回答中。

### 4.5 Python 文档解析 Sidecar

OCR、复杂文档解析和表格深度理解不塞进 Rust 主进程，而是通过 Python Sidecar 执行。

Python Sidecar 负责：

- 本地 OCR
- 扫描 PDF 和图片解析
- Word/PDF/XLSX/Markdown 内容抽取
- 表格结构还原
- 文档版面和阅读顺序识别
- GPU/CPU 分层模型调用

Rust 负责启动、停止、监控 Sidecar，并通过本地进程通信传递任务。Sidecar 只处理明确授权的本地文件，不直接修改知识库文件结构。

### 4.6 云端模型边界

云端模型第一版仅接入 DeepSeek `deepseek-v4-flash`。

DeepSeek 负责：

- 摘要生成
- 标签建议
- 知识块标题生成
- 表格字段语义说明
- Agent 问答生成

DeepSeek 不负责永久存储知识库数据。发送到云端的内容由 Rust 核心按当前范围、权限和敏感设置组装。敏感文件夹可启用上传前确认。

## 5. 主界面和信息架构

主界面采用中文三栏工作台：

- 左侧：知识库文件夹树，显示真实知识空间和默认权限
- 中间：当前文件夹工作区，包含总览、文件、知识块、表格、回收站
- 右侧：智能助手聊天栏，显示当前范围和当前会话权限

当前视觉方向已经确认：界面简约、克制、偏工具型。标题区保持轻量，一行展示当前文件夹和真实路径，例如 `面试  D:\知识库\面试`。状态信息不使用抢眼胶囊，降级放在内容区顶部。

## 6. 数据流和解析管线

导入一个文件夹后，应用先扫描真实磁盘结构，只记录路径、哈希、修改时间和文件类型，不改变原文件。

解析流程：

1. 文件识别：判断文件是 Word、PDF、XLSX、Markdown、图片、扫描 PDF，决定解析策略
2. 内容抽取：可复制文本直接抽取，扫描文档和图片进入本地文字识别队列
3. 结构解析：提取页面、标题、段落、阅读顺序、表格、图片区域
4. 语义增强：调用 DeepSeek `deepseek-v4-flash` 生成摘要、标签、知识块标题、表格字段说明
5. 索引入库：保存 Markdown 笔记、知识块、来源位置、表格 JSON/HTML、向量索引

每个原文件会产生两类结果：

- 同名 Markdown 笔记：放在真实文件夹中，用户可自由编辑
- 知识块索引：保存在本地数据库中，用于搜索、问答、引用和删除

每个文件夹还会生成一个 `README.md` 总览，汇总该文件夹的核心知识、文件清单和复习主题。

当原文件变化时，默认只提示，不自动重新解析。用户点击重新解析后，系统不覆盖用户修改过的 Markdown，而是生成更新建议或新版本草稿。内部索引则基于文件哈希和来源位置重建。

## 7. 权限模型和 Agent 操作规则

权限分为两层：

- 文件夹默认权限：长期配置，保存在本地数据库
- 当前聊天权限：只作用于本次会话，不写回文件夹默认设置

权限档位：

- 只读：智能助手只能检索、总结、引用来源、生成未保存草稿
- 请求批准：智能助手可以提出创建笔记、保存草稿、修改标签、移动知识块、删除知识块等操作，但执行前必须等待用户批准
- 完全访问：第一版提供该档位，但默认关闭，需要用户显式开启；删除、覆盖、批量移动、清空回收站等高风险操作仍强制确认

权限和范围必须分离：

- 范围决定智能助手能读取哪里，例如当前文件、当前文件夹、全库
- 权限决定智能助手能做什么，例如只读、请求批准

高风险操作不允许被普通权限绕过。删除原文件、清空回收站、批量移动、覆盖 Markdown、重新解析大批文件、发送敏感文件内容到云端模型，都必须显示影响范围并二次确认。

## 8. 本地文字识别和表格深度理解

文字识别采用本地分层策略：

- 有 NVIDIA GPU：使用高质量本地 OCR/文档解析模型管线
- 无 GPU：使用 CPU 或轻量模式，任务进入后台队列
- 所有情况：主界面不被阻塞，任务面板显示进度、预计耗时和失败原因

本地 OCR 负责原始识别和版面结构提取。云端 DeepSeek `deepseek-v4-flash` 只负责语义增强和问答。

表格深度理解第一版包含：

- 识别工作表、表头、合并单元格、行列关系
- 识别字段含义、单位、日期列、分类列、指标列
- 生成表格摘要和标签
- 支持自然语言问答，例如最高值、最低值、趋势、按类别汇总、筛选比较

第一版不做复杂报表仪表盘、透视表编辑器和高级数据清洗。

候选技术资料需要在实现前再次复核官方文档。当前设计调研参考：

- PaddleOCR：https://github.com/PaddlePaddle/PaddleOCR

## 9. 删除、回收站和一致性

第一版只做简单回收站机制，不做完整版本管理。

删除类型：

- 删除原文件：默认移入应用回收站或系统回收站，不直接永久删除
- 删除 Markdown 笔记：移入回收站，同时标记相关索引不可用
- 删除知识块：不改原文件，只让该知识块不再参与搜索、问答和引用

知识块删除必须在界面上说明影响范围。例如：此操作会让该知识块不再出现在搜索和回答中，原文件不会被修改。

回收站按文件夹隔离展示，避免跨知识库误恢复。恢复时，系统恢复文件、笔记或知识块状态，并把相关索引加入重建队列。清空回收站是高风险操作，必须二次确认。

一致性原则：

- 真实文件夹优先
- 索引可重建
- 如果磁盘文件和索引不一致，界面显示需要重新索引，不静默猜测

## 10. 备份、导入导出和未来同步预留

第一版不做账号系统和多设备实时同步，但支持完整备份迁移。

备份包包含：

- 真实文件夹里的原始文件和 Markdown 笔记
- 本地数据库中的知识块、权限、回收站、解析任务历史
- 表格结构化结果、向量索引和聊天引用记录
- 应用配置，例如 DeepSeek 模型配置、文字识别分层策略、文件夹默认权限

导入时，应用先扫描备份包，显示包含哪些知识库、文件数量、数据库版本和潜在冲突，再让用户确认。遇到同名知识库时，提供合并、改名导入、跳过三种选择，不默认覆盖。

未来同步预留字段：

- 稳定 ID
- 创建时间和更新时间
- 删除标记
- 设备来源
- 冲突状态
- 最后解析版本

这些字段第一版主要用于未来兼容，不在界面暴露复杂同步功能。

## 12. 测试重点

- 文件夹扫描、改名、移动、删除后的索引一致性
- Word、PDF、XLSX、Markdown 的解析质量
- 扫描 PDF 和图片文字识别的 GPU/CPU 分层表现
- 表格问答的准确性和失败提示
- 文件夹默认权限和聊天临时权限隔离
- 删除、恢复、清空回收站的影响范围
- Markdown 被用户修改后，不被重新解析覆盖
- DeepSeek 调用失败、网络断开、余额不足、上下文超长时的降级体验
- SQLite 与 LanceDB 在删除、恢复、重新索引后的状态一致性
- Python Sidecar 启动失败、模型缺失、任务超时、GPU 不可用时的降级体验
- 备份导入导出和同名知识库冲突处理

## 13. 主要风险

1. 本地文字识别和表格模型部署复杂，可能受显卡、驱动、Python 环境影响。
2. 真实文件夹和本地索引的一致性是长期可靠性的关键，需要优先设计任务队列和重建机制。
3. 权限和删除操作如果表达不清，会影响用户信任，因此每个高风险动作都必须展示影响范围。
4. Markdown 允许用户自由编辑后，重新解析必须避免覆盖用户内容。
5. DeepSeek 云端调用需要明确上传范围，敏感文件夹应支持上传前确认。
6. SQLite 和 LanceDB 是双数据库，本地 ID、删除状态和重建任务必须保持一致。
7. Python Sidecar 与 Tauri/Rust 主进程的打包、启动和错误恢复需要尽早验证。

## 15. 前端设计原则

- 界面默认中文
- 三栏布局保持稳定，不做花哨动效
- 标题区简约，路径和状态降级显示
- 操作按钮和风险提示必须清楚
- 不用过度装饰、发光渐变、卡片堆叠
- 控件状态完整，包括空状态、加载、失败、禁用、待批准
- 颜色、间距、圆角、阴影通过设计变量统一管理
- 具体页面和组件样式放在组件作用域内，不把业务样式堆进全局 CSS
